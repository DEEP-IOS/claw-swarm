**Claw-Swarm V9.0.0** | 8D 模型画像与 MoE 路由

## 设计理由

并非所有 LLM 模型都是等价的。一个擅长代码生成的模型可能在架构推理上表现不佳；一个快速、低成本的模型可能完美胜任文档编写，却不适合安全分析。传统的多代理系统要么硬编码单一模型，要么将模型选择完全交给用户——两者都无法扩展。

模型能力注册表通过以下机制解决这一问题：

1. **画像构建** -- 在 8 个正交维度上为每个模型评分，分数由权威基准测试的加权组合导出
2. **实时匹配** -- 将任务的 8D 需求向量与模型的 8D 能力向量做点积计算，选择最佳匹配
3. **持续校准** -- 根据实际任务结果通过 EMA（指数移动平均）持续修正模型分数，适应现实漂移

这使蜂群无需人工干预即可为每个任务阶段自动选择最优模型。

## 8D 能力维度

每个模型在 8 个维度上评分，每个维度由已建立的基准测试的加权组合导出。分数归一化至 [0, 1] 区间。

### 1. Coding（编程能力）

衡量代码生成、补全和 bug 修复能力。

```
coding = 0.25 * HumanEval + 0.35 * SWE-bench + 0.25 * LiveCodeBench + 0.15 * MATH-500
```

**权重设计说明：** SWE-bench 权重最高（0.35），因为它测试真实代码库中的修复能力，比孤立的函数生成更能代表实际工程能力。MATH-500 作为辅助维度（0.15），因为复杂编程任务通常包含算法推理。

### 2. Architecture（架构能力）

衡量系统设计、复杂结构推理和抽象问题解决能力。

```
architecture = 0.35 * GPQA-Diamond + 0.30 * MATH-500 + 0.20 * MMLU-Pro + 0.15 * MMLU
```

**权重设计说明：** GPQA-Diamond（研究生级别 QA）权重最高，因为架构推理需要深度领域专业知识而非广泛常识。

### 3. Testing（测试能力）

衡量测试生成、bug 复现和质量保障能力。

```
testing = 0.45 * SWE-bench + 0.25 * LiveCodeBench + 0.15 * HumanEval + 0.15 * IFEval
```

**权重设计说明：** SWE-bench 占主导（0.45），因为测试能力的核心是理解现有代码库并生成有效测试，而非编写独立函数。

### 4. Documentation（文档能力）

衡量指令遵循、清晰表达和知识广度。

```
documentation = 0.40 * IFEval + 0.25 * Arena-Hard + 0.20 * MMLU + 0.15 * MMLU-Pro
```

**权重设计说明：** IFEval（指令遵循评估）权重最高，因为文档编写的核心是严格遵循格式和结构要求。

### 5. Security（安全能力）

衡量安全意识、抗幻觉能力和行为一致性。

```
security = 0.30 * IFEval + 0.25 * MMLU-Pro + 0.25 * hallucination_inv + 0.20 * consistency
```

其中 `hallucination_inv` 为幻觉率的倒数归一化值，`consistency` 为多次运行输出一致性。

**权重设计说明：** 安全维度是唯一包含非公开基准指标的维度。幻觉率和一致性通过内部评测获得，因为安全相关任务对可靠性要求极高。

### 6. Performance（性能效率）

衡量成本效率、速度和上下文利用率。

```
performance = normalize(cost_inv) + normalize(speed) + normalize(contextEfficiency)
```

其中 `cost_inv` 为每百万 token 成本的倒数，`speed` 为每秒输出 token 数，`contextEfficiency` 为有效上下文利用率。

**权重设计说明：** 三指标等权，因为不同场景下用户对成本、速度、长文本处理的优先级各异。

### 7. Communication（沟通能力）

衡量对话质量、说服力和人类偏好对齐度。

```
communication = 0.40 * Arena-Hard + 0.30 * IFEval + 0.15 * MMLU + 0.15 * ELO
```

**权重设计说明：** Arena-Hard（人类偏好竞技场）权重最高，因为沟通质量本质上由人类主观评判定义。

### 8. Domain（领域专业度）

衡量跨学术和职业领域的专业化知识。

```
domain = 0.30 * MMLU + 0.25 * MMLU-Pro + 0.20 * C-Eval/CMMLU + 0.15 * GPQA + 0.10 * specialized
```

其中 `C-Eval/CMMLU` 覆盖中文领域知识，`specialized` 为针对特定行业的评测。

**权重设计说明：** MMLU 系列占主导（0.55），因为广泛的领域知识是专业任务的基础。C-Eval/CMMLU 确保中文领域不被忽视。

### 画像解读

**旗舰模型（claude-opus-4-6）** 在除 Performance 外的所有维度上均达到 0.90+，但性能维度仅 0.60——反映了大模型的高成本和较低速度。适合高质量需求、对成本不敏感的任务。

**性价比模型（claude-haiku-4-5, deepseek-r1）** Performance 维度显著领先（0.85-0.92），但在架构推理和沟通质量上有所折衷。适合批量处理、成本敏感或时间紧迫的任务。

**全能型模型（claude-sonnet-4-6, gpt-4.1）** 各维度均衡，无明显短板。适合需要多维度能力的综合任务。

## MoE 路由机制

当 `swarm_run` 规划任务执行时，MoE 路由器按以下步骤运行：

### 步骤 1: 推断任务需求

调用低成本 LLM 分析任务目标，生成 8D 需求向量。

```
taskVector = llmAnalyzeTask(goal)
// 返回: [coding: 0.9, architecture: 0.3, testing: 0.8, ...]
```

回退机制：当 LLM 分析不可用时，使用基于关键词的 `inferRequirements()` 方法。关键词映射规则：

- 含 "代码/实现/编程" -> coding 维度提升
- 含 "设计/架构/系统" -> architecture 维度提升
- 含 "测试/验证/QA" -> testing 维度提升
- 含 "文档/README/说明" -> documentation 维度提升
- 含 "安全/漏洞/审计" -> security 维度提升

### 步骤 2: 计算相似度

对每个候选模型，计算其 8D 能力向量与任务需求向量的点积：

```
modelScore = sum( taskVector[d] * modelCapability[d] )  对所有维度 d
```

这是标准的点积相似度，本质上是余弦相似度的非归一化版本（因为能力向量和需求向量的模长已经隐含在分数大小中）。

### 步骤 3: 选择模型

```
selectedModel = argmax(modelScore)
```

平局打破规则：当多个模型得分相同时，优先选择 Performance 维度更高的模型（更低成本、更快速度）。

### 步骤 4: 分配至子代理

执行计划中的每个阶段可以使用不同的最优模型。例如：

```
阶段 1（架构设计）-> claude-opus-4-6    (architecture: 0.95)
阶段 2（代码实现）-> claude-sonnet-4-6  (coding: 0.88, performance: 0.75)
阶段 3（测试编写）-> gpt-4.1           (testing: 0.85, performance: 0.70)
阶段 4（文档生成）-> claude-haiku-4-5   (documentation: 0.78, performance: 0.92)
```

这种按阶段异构分配是 MoE 路由的核心价值：每个阶段使用该阶段需求维度上最强的模型，而非对所有阶段使用同一模型。

## EMA 校准机制

模型分数不是静态的。每次任务完成后，系统观测：

- 任务是否成功？
- 质量评分是多少？
- 耗时多长？

这些观测通过 EMA（指数移动平均）更新模型能力分数：

```
newScore[d] = alpha * observedScore[d] + (1 - alpha) * oldScore[d]
```

### 参数说明

| 参数 | 值 | 含义 |
|------|-----|------|
| `alpha` | 0.3 | 平滑因子：近期观测占 30% 权重，历史分数占 70% 权重 |
| `minSamples` | 5 | 最小样本数：低于此阈值不触发校准（防止冷启动噪声） |

### 校准行为

- **持续低于预期的维度** -- 分数逐渐衰减。例如，某模型在 testing 维度上连续产出低质量测试，其 testing 分数会从初始值缓慢下降。
- **超出预期的维度** -- 分数逐渐上升。例如，某模型在实际任务中展现出超越基准测试的编程能力，其 coding 分数会逐步提高。
- **alpha = 0.3 的含义** -- 单次异常观测对总分的影响有限（仅 30%），需要多次一致观测才能显著改变分数。这防止了单次运气好/坏的任务结果大幅扭曲画像。

### 衰减示例

假设 claude-haiku-4-5 的 coding 初始分为 0.75，连续三次编程任务分别观测到 0.60, 0.55, 0.58：

```
第 1 次: 0.3 * 0.60 + 0.7 * 0.75 = 0.705
第 2 次: 0.3 * 0.55 + 0.7 * 0.705 = 0.659
第 3 次: 0.3 * 0.58 + 0.7 * 0.659 = 0.635
```

三次观测后，coding 分数从 0.75 下降到 0.635，系统会自动将编程任务路由到其他模型。

## ModuleBase 集成

`ModelCapabilityRegistry` 继承 `ModuleBase`，接入 12 维信号场：

| 属性 | 值 | 说明 |
|------|-----|------|
| **Address** | `/engine/model-registry` | 信号场中的唯一地址 |
| **Receptor** | `model_update` | 入站信号：当基准数据或 EMA 观测到达时触发重新计算 |
| **Effector** | `model_scores_updated` | 出站信号：能力画像变化时通知下游系统 |

### 信号流

```
[观测数据] --model_update--> [ModelCapabilityRegistry]
                                      |
                              EMA 重新计算
                                      |
                      --model_scores_updated--> [DualProcessRouter]
                      --model_scores_updated--> [ExecutionPlanner]
                      --model_scores_updated--> [ABCScheduler]
```

当模型画像发生变化时，下游的路由器和规划器自动接收更新信号，无需轮询或手动刷新。

## 配置参考

注册表开箱即用，无需配置——35+ 模型画像全部内置。自定义模型会根据最接近的内置模型家族自动生成初始画像。

### 可选覆盖

在 `openclaw.json` 中可配置以下参数：

```json
{
  "plugins": {
    "entries": {
      "claw-swarm": {
        "config": {
          "modelRegistry": {
            "emaAlpha": 0.3,
            "minSamples": 5
          }
        }
      }
    }
  }
}
```

| 参数 | 默认值 | 说明 |
|------|--------|------|
| `emaAlpha` | 0.3 | EMA 平滑因子。增大使模型更快适应观测变化，但也更不稳定 |
| `minSamples` | 5 | 触发 EMA 校准的最小观测样本数。增大可降低冷启动噪声 |

### 自定义模型注册

当蜂群使用未在内置列表中的模型时，注册表自动执行以下流程：

1. 解析模型 ID 前缀，匹配最接近的模型家族（如 `gpt-4o-mini` 匹配 `gpt-4o` 家族）
2. 复制家族基准画像作为初始值
3. 根据已知的成本/速度差异调整 Performance 维度
4. 后续通过 EMA 机制持续校准

## 设计权衡与局限

### 为何选择 8 维而非更多/更少？

- **少于 8 维** -- 无法区分模型间的细粒度差异（例如 coding 和 testing 高度相关但并非完全等价）
- **多于 8 维** -- 维度间相关性增加，点积路由的区分力下降（维度灾难）
- **8 维** -- 覆盖了 LLM 代理的主要任务类型，各维度之间保持合理的正交性

### 点积 vs 余弦相似度

选择点积而非余弦相似度，因为向量的**模长包含信息**：一个全维度 0.9 的模型（模长大）应该被优先选择，而余弦相似度会抹平这一差异。

### EMA vs 贝叶斯更新

EMA 实现简单、内存开销恒定（O(1) 每模型每维度），而贝叶斯后验更新需要维护完整的分布参数。对于 35+ 模型 x 8 维度 = 280 个参数的规模，EMA 的简洁性更为合适。
